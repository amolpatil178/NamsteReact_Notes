What is NPM?

The npm is the world's largest software registry. Generally it manages the packages and modules for Node.js. It's all code is written in JavaScript.The npm is installed when we install node.js application. It has two things, a CLI command line interface to for publishing and downloading packages and an online repository that hosts JavaScript packages. Each package has it's own package.json file. So whenever we initialiaze npm by command npm init to run our project as npm package. It will create package.json file. package.json file is configuration file of that package. It will contain :

  1. name: name of JavaScript package/project
  2. version: the version of your project.
  3. description: the project's description
  4. license: the project's license
  5. It also contains scripts property that can be defined to run command-line tools that are installed in the project's local context. e.g. format,lint,jest etc
  6. It also contains depedencies and DevDependencies

What is "parcel/webpack"? why do we need it?

The parcel and webpack are bundlers. The bundler is a tool where it bundles all you javascript files and all depedencies in to single file. In React, bundlers are used to combine all the JavaScript files that make up a React application into a single file that can be loaded by a web browser. This is done to reduce the number of HTTP requests that a browser has to make to load a web page, which can significantly improve the performance of a React application. Bundlers in React can also be used to perform other tasks, such as transpiling code from newer versions of JavaScript to older versions that are supported by all browsers, and optimizing images and other assets

What is "parcel-cache"?

The .cache folder (or .parcel-cache in parcel v2) stores information about your project when parcel builds it, so that when it rebuilds, it doesn't have to re-parse and re-analyze everything from scratch. It's a key reason why parcel can be so fast in development mode.

what is deifference between "DevDependencies" vs "depedencies"?

In every web application project, we have a file called package.json. This file contains all the relevant data regarding the project i.e. metadata. Starting from all the dependencies used to all the version numbers are present in the file.

depedencies : In package.json file, there is an object called dependencies and it consists of all the packages that are used in the project with its version number. So, whenever you install any library that is required in your project that library you can find it in the dependencies object. Generally it is used for production enviornment.
e.g.npm install package_name then after installing, you will see entry of this package name with version in dependencies object in package.json file.

DevDependencies : In package.json file, there is an object called as dev Dependencies and it consists of all the packages that are used in the project in its development phase and not in the production or testing environment with its version number. So, whenever you want to install any library that is required only in your development phase then you can find it in the dev Dependencies object.
e.g.npm install package_name --save-dev then after installing, you will see entry of this package name with version in devDependencies object in package.json file.

what is tree shaking?

Tree shaking is a term commonly used within a JavaScript context to describe the removal of dead code. It relies on the import and export statements to detect if code modules are exported and imported for use between JavaScript files. In modern JavaScript applications, we use module bundlers to automatically remove dead code when bundling multiple JavaScript files into single files. This is important for preparing code that is production ready, for example with clean structures and minimal file size. Parcel also statically analyzes the imports and exports of each module, and removes everything that isn't used. This is called "tree shaking" or "dead code elimination". Tree shaking is supported for both static and dynamic import, CommonJS and ES modules, and even across languages with CSS modules.

What is Hot module Replacement?

Hot Module Replacement improves the development experience by updating modules in the browser at runtime without needing a whole page refresh. This means that application state can be retained as you change small things in your code. The CSS changes are automatically applied via HMR with no page reload necessary. This is also true when using a framework with HMR support built in, like React (via Fast Refresh), and Vue.

What is .gitignore? What should we add and not add into it ?

A gitignore file specifies intentionally untracked files that Git should ignore. A gitignore is the file where we add the files and folder which are not supposed to be pushed into the git repository. E.g., the files which can be generated again from other files (e.g. node_modules, dist, .parcel-cache, etc.) are not required to be pushed into the git repo.

A package.json file contains metadata about the project and also the functional dependencies that is required by the application. A package-lock.json is created for locking the dependency with the installed version. It will install the exact latest version of that package in your application and save it in package.json.

What is node_modules? Is it good idea to push that to git?

node_modules is a directory in Node.js projects that stores external libraries and modules. It is not advisable to push it to Git because of its size and redundancy with package files. What are vite, webpack and parcel?

The vite, webpack and parcel are bundlers. These tools cater to different project needs and developer preferences, with Vite focusing on speed and efficiency, Webpack offering flexibility and customization, and Parcel prioritizing simplicity and ease of use.

Key Vite Webpack Parcel
Purpose Designed for fast modern web development. Highly configurable build tool suitable for complex projects. "Zero-config" build tool for simple project setups.
Development Server Provides a fast development server with native ES modules (ESM) for rapid hot module replacement (HMR). Offers a development server and HMR support but requires more setup compared to Vite. Includes a development server with built-in HMR for real-time changes.
Build Process Uses ESM during development and generates optimized bundles for production. Provides extensive control over code splitting, asset optimization, and more. Automatically detects and configures project settings, optimizing and bundling assets.
Configuration Supports both zero-config and manual configuration. Relies on a detailed configuration file (webpack.config.js) for fine-tuning. Aims to minimize configuration, making it suitable for smaller projects and rapid development.

what is ^ caret and ~ tilda in

Caret (^): The caret (^) allows compatible updates but restricts breaking changes.
Example: "dependencies": { "package-name": "^1.2.3" } permits versions from 1.2.3 up to, but not including, 2.0.0.

Tilde (~): The tilde (~) allows updates to patch versions but restricts minor and major changes.
Example: "dependencies": { "package-name": "~1.2.3" } allows versions from 1.2.3 up to, but not including, 1.3.0.

Read About Script types in HTML ?

Inline scripts : These are JavaScript code blocks placed directly within the HTML document, enclosed in script tags. They can be in the head or body sections.
External script : These are JavaScript files linked to the HTML document using the script tag src attribute. They can be in the head or body sections.

In HTML, the type attribute in the script tag specifies the content type of the script being included. Here are the commonly used type values: text/javascript: Specifies that the content is JavaScript code. This type is widely supported and used by default if the type attribute is omitted.
module: Indicates that the script is a JavaScript module, allowing for modular code organization with features like import and export.
application/json: Defines JSON data. Not commonly used for scripts but rather for embedding JSON data within the HTML document.
text/html: Rarely used for scripts; typically used to embed HTML code as a script.
text/css: Specifies that the content is CSS code. Rarely used in modern development due to the separation of CSS in separate files.
text/plain: Specifies plain text, not treated as executable code.
